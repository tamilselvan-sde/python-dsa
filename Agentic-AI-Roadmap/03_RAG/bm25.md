# BM25 Algorithm

## 1. What is it?
**ELI5:** BM25 is like a smart scoring system for search results. Imagine you're looking for a recipe with "chocolate cake" in a cookbook. BM25 thinks: "How many times does 'chocolate' appear in each recipe? Is that a lot compared to other recipes? Is the recipe unusually long (which might inflate the counts)? And is 'chocolate' a rare word (which makes it more important)?" It combines all these signals into a single relevance score.

**Technical Definition:** BM25 (Best Matching 25) is a bag-of-words retrieval function that ranks documents by relevance to a query. It computes a score as the sum over query terms of term frequency saturation (controlled by k1), document length normalization (controlled by b), and inverse document frequency. It's the state-of-the-art sparse retrieval algorithm and the default scoring function in Elasticsearch, Lucene, and Solr.

## 2. Why do we need it?
- **State-of-the-art sparse retrieval:** Outperforms TF-IDF on every standard IR benchmark (TREC, MS MARCO).
- **Controlled term saturation:** BM25's k1 parameter prevents a term appearing 100 times from scoring 100x higher than appearing once — diminishing returns match human relevance judgment.
- **Length normalization:** Longer documents naturally have higher term frequencies. BM25's b parameter compensates, preventing long documents from dominating results.
- **Production standard:** Elasticsearch, Lucene, Solr, and Amazon CloudSearch all use BM25 as default. It's the most deployed retrieval algorithm in the world.
- **Zero training required:** Unlike learned retrieval, BM25 works perfectly out of the box with any text corpus.

## 3. Real-world Example
- **Elasticsearch (all deployments):** Every Elasticsearch cluster uses BM25 by default. Wikipedia's internal search, Stack Overflow's "similar questions" feature, and GitHub's code search all run on BM25.
- **Elastic's Relevance Tuning:** Elasticsearch exposes k1 and b for tuning. E-commerce search often uses lower k1 (less saturation — every mention matters), while legal search uses higher k1 (first mention is much more important than subsequent).
- **Lucene (underlying most Java search):** Apache Lucene's BM25 implementation powers Adobe's AEM search, LinkedIn's search, and Twitter's early search infrastructure.
- **Tantivy (Rust):** Rust-based BM25 implementation used by Quickwit and Meilisearch for log analytics and full-text search.

## 5. Internal Working
BM25 score for a document D given query Q (terms q1...qn):

```
score(D, Q) = Σ_i IDF(qi) × f(qi, D) × (k1 + 1) / [f(qi, D) + k1 × (1 - b + b × |D| / avgdl)]
```

**Components:**
- **f(qi, D):** Term frequency — how many times term qi appears in document D
- **|D|:** Document length (total number of terms)
- **avgdl:** Average document length across the entire corpus
- **k1 (1.2-2.0):** Term frequency saturation parameter. Controls how quickly TF contributions plateau. Default: 1.5
- **b (0.0-1.0):** Length normalization parameter. b=0: no normalization. b=1: full normalization. Default: 0.75
- **IDF(qi):** Inverse document frequency: `log((N - df + 0.5) / (df + 0.5) + 1)` where N = total docs, df = docs containing qi

**Key behaviors:**
1. A term appearing 10 times is NOT 10x better than appearing once (k1 saturation)
2. A 100-word document mentioning a term once is more relevant than a 1000-word document mentioning it once (b normalization)
3. Rare terms contribute more to the score than common terms (IDF weighting)

## 8. LLD
```python
import math
from collections import Counter, defaultdict
from typing import Dict, List, Tuple

class BM25:
    def __init__(self, k1: float = 1.5, b: float = 0.75):
        self.k1 = k1
        self.b = b
        self.index: Dict[str, List[Tuple[str, int]]] = defaultdict(list)
        self.doc_lengths: Dict[str, int] = {}
        self.doc_count: int = 0
        self.avgdl: float = 0.0
        self.doc_store: Dict[str, str] = {}

    def add_document(self, doc_id: str, tokens: List[str], text: str = ""):
        tf = Counter(tokens)
        self.doc_lengths[doc_id] = len(tokens)
        self.doc_store[doc_id] = text or doc_id
        old_avg = self.avgdl
        self.doc_count += 1
        self.avgdl = old_avg + (len(tokens) - old_avg) / self.doc_count
        for term, freq in tf.items():
            self.index[term].append((doc_id, freq))

    def idf(self, term: str) -> float:
        if term not in self.index:
            return 0.0
        df = len(self.index[term])
        return math.log((self.doc_count - df + 0.5) / (df + 0.5) + 1)

    def score_doc(self, doc_id: str, query_terms: List[str]) -> float:
        score = 0.0
        dl = self.doc_lengths.get(doc_id, 1)
        for term in query_terms:
            if term not in self.index:
                continue
            tf = 0
            for pid, ptf in self.index[term]:
                if pid == doc_id:
                    tf = ptf
                    break
            if tf == 0:
                continue
            idf = self.idf(term)
            numerator = tf * (self.k1 + 1)
            denominator = tf + self.k1 * (1 - self.b + self.b * dl / self.avgdl)
            score += idf * numerator / denominator
        return score

    def search(self, query_terms: List[str], top_k: int = 10) -> List[Tuple[str, float]]:
        candidate_docs = set()
        for term in query_terms:
            if term in self.index:
                for doc_id, _ in self.index[term]:
                    candidate_docs.add(doc_id)

        scored = [(doc_id, self.score_doc(doc_id, query_terms)) for doc_id in candidate_docs]
        scored.sort(key=lambda x: x[1], reverse=True)
        return scored[:top_k]
```

## 9. Python Implementation
```python
import math
import re
from collections import Counter, defaultdict
from typing import List, Tuple

import numpy as np
from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI(title="BM25 Search Service")

class BM25Query(BaseModel):
    query: str = Field(..., min_length=1)
    top_k: int = Field(default=10, ge=1, le=100)
    k1: float = Field(default=1.5, ge=0.0, le=3.0)
    b: float = Field(default=0.75, ge=0.0, le=1.0)

class BM25Result(BaseModel):
    doc_id: str
    score: float
    text: str

class BM25Response(BaseModel):
    results: List[BM25Result]
    params: dict
    latency_ms: float

def tokenize(text: str) -> List[str]:
    return re.findall(r"\w+", text.lower())

class BM25Engine:
    def __init__(self):
        self.index: dict = defaultdict(list)
        self.doc_lengths: dict = {}
        self.doc_store: dict = {}
        self.num_docs = 0
        self.avgdl = 0.0

    def add(self, doc_id: str, text: str):
        tokens = tokenize(text)
        self.doc_store[doc_id] = text
        self.doc_lengths[doc_id] = len(tokens)
        self.num_docs += 1
        self.avgdl += (len(tokens) - self.avgdl) / self.num_docs
        for term, freq in Counter(tokens).items():
            self.index[term].append((doc_id, freq))

    def search(self, query: str, top_k: int = 10, k1: float = 1.5, b: float = 0.75) -> list:
        tokens = tokenize(query)
        scores = defaultdict(float)

        for term in set(tokens):
            if term not in self.index:
                continue
            df = len(self.index[term])
            idf = math.log((self.num_docs - df + 0.5) / (df + 0.5) + 1)
            for doc_id, tf in self.index[term]:
                dl = self.doc_lengths[doc_id]
                tf_norm = tf * (k1 + 1) / (tf + k1 * (1 - b + b * dl / self.avgdl))
                scores[doc_id] += idf * tf_norm

        ranked = sorted(scores.items(), key=lambda x: x[1], reverse=True)[:top_k]
        return [
            BM25Result(doc_id=did, score=score, text=self.doc_store.get(did, ""))
            for did, score in ranked
        ]

engine = BM25Engine()

for i in range(1000):
    engine.add(f"doc_{i}", f"Document about machine learning and AI #{i}")

@app.post("/bm25/search", response_model=BM25Response)
async def bm25_search(q: BM25Query):
    import time
    start = time.time()
    results = engine.search(q.query, q.top_k, q.k1, q.b)
    return BM25Response(
        results=results,
        params={"k1": q.k1, "b": q.b},
        latency_ms=(time.time() - start) * 1000,
    )
```

## 14. Pros
- Best-in-class sparse retrieval — outperforms TF-IDF on all benchmarks
- Configurable (k1, b) — tunable per corpus and domain
- Works without any training data or ML infrastructure
- Fast — millisecond latency on millions of documents
- Robust — proven at Google, Netflix, LinkedIn scale
- Interpretable — score decomposition shows why each doc matched

## 15. Cons
- No semantic understanding — vocabulary mismatch is the primary failure mode
- Parameters (k1, b) require tuning per corpus
- Degrades on very short documents (tweets, titles)
- Doesn't handle multi-word expressions well without positions
- No cross-lingual capability — query must be in document language

## 16. Alternatives
| Algorithm | Description |
|-----------|-------------|
| TF-IDF | Simpler, no saturation or length normalization |
| BM25F | Variant for structured documents (multiple fields) |
| BM25+ | Adds a lower-bound term frequency component |
| SPLADE | Learned sparse retrieval with neural term weighting |
| Dense retrieval | Neural embedding-based semantic search |
| ColBERT | Token-level interaction matching |

## 17. Performance Considerations
- **Index size:** ~50% of original corpus (compressed posting lists)
- **Search time:** O(query_terms × avg_posting_length). Typically <5ms for 1M docs.
- **Memory:** mmap-based indexes scale to billions of docs without filling RAM.
- **Tuning grid:** k1 ∈ [0.5, 2.0] step 0.3, b ∈ [0.3, 0.9] step 0.1. Measure NDCG@10.
- **Pre-compute IDF** for performance — don't recompute on every query.

## 18. Scaling to Millions
- **Shard by doc_id hash** across Elasticsearch cluster nodes.
- **Replicate shards** for read throughput.
- **Use tiered indexing:** Hot (SSD, recent), warm (HDD, less recent), cold (object store).
- **Compress posting lists** with variable-byte encoding — 3-5x compression.
- **Cache top-1000 queries** in Redis for sub-millisecond response.

## 19. Failure Scenarios
| Failure | Impact | Mitigation |
|---------|--------|------------|
| Zero IDF for common terms | All docs score similarly | Remove high-DF terms |
| Degenerate avgdl with one doc | Length normalization broken | Seed with reasonable default |
| Memory exhaustion | Index unavailable | Use mmap, set memory limit |
| Tokenizer mismatch | No results | Use exact same tokenizer for indexing and querying |

## 20. Security
- Validate and sanitize index content before ingestion.
- Apply document-level security (filter results based on user permissions).
- Rate-limit search endpoint.
- Log all queries for audit trail.

## 21. Monitoring
- **Average BM25 score:** Track score distribution — drift indicates content changes.
- **Query latency:** p50, p95, p99 per shard.
- **Zero-result rate:** % of queries returning 0 results.
- **Index size:** Disk usage per shard.
- **k1/b parameter usage:** What values perform best per query type.

## 24. Common Mistakes
- **Using default k1=1.5, b=0.75 for every corpus** — tune these! Legal documents need different settings than tweets.
- **Not removing stopwords** — "the" and "and" contribute noise and slow down searches.
- **Token mismatch between index and query** — if you stemmed during indexing, you MUST stem during query.
- **Setting b=0** — long documents will dominate all results (no length normalization).
- **Ignoring k1 when documents vary wildly in length** — if short + long docs, tune k1 and b together.
- **Using BM25 for semantic matching** — BM25 is lexical; pair it with dense retrieval for semantic needs.

## 25. Production Best Practices
1. **Always tune k1 and b** — run a grid search on a held-out relevance set. Start with k1=1.2, b=0.75.
2. **Use the same tokenizer** for indexing and querying — different tokenization breaks retrieval silently.
3. **Pre-compute IDF values** — store them in a hash table for O(1) lookup during query time.
4. **Implement early termination** — stop scoring documents after a certain threshold to bound latency.
5. **Monitor query term coverage** — if many query terms have no IDF (missing from index), expand vocabulary or use hybrid search.
6. **Cache hot queries** — the Pareto principle applies: 20% of queries get 80% of traffic.
7. **Combine with dense retrieval** — BM25 + embeddings in a hybrid fusion is strictly better than either alone.
8. **Log relevance judgments** — collect implicit feedback (click-through rates) to tune BM25 parameters continuously.
